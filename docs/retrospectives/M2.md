## Scope delivered

Plan §7.3's three-week task list, exit criteria all hold:

### Week 1 — apply-side adapters

- **`pearlite-snapper`** (Plan §6.9, PR #21): `Snapper` trait +
  `LiveSnapper` + `MockSnapper`; `create` / `rollback` / `list`;
  deterministic snapshot label generation; `snapper list` parser.
- **`pearlite-pacman`** apply methods (PR #22): `sync_databases`,
  `install(repo, &[pkg])`, `aur_install`, `remove`. Mock exposes
  `*_history()` inspectors for engine integration tests.
- **`pearlite-cargo`** apply methods (PR #23): `install(name, locked)`,
  `uninstall`. Same mock-history pattern.
- **`pearlite-systemd`** apply methods (PR #24): `enable` / `disable` /
  `mask` / `restart` with a local `Scope` enum (System | User { name })
  that maps to `runuser -u <name> -- systemctl --user`.
- **ADR-008** + **ADR-009** (PR #20): MSRV bump policy and CachyOS CI
  runner choice. M1 carry-over debt cleared.

### Week 2 — engine orchestration

- **`pearlite-diff`** classifiers extended (PRs #25 / #26):
  `Action::failure_coherence` returns `Recoverable` (Class 3) or
  `Incoherent` (Class 4) per PRD §8.5; `Action::phase` returns the
  PRD §8.2 `ApplyPhase` discriminant. Pure data classifiers.
- **`pearlite-engine::apply_plan`** (PRs #27 – #30): seven-phase
  orchestrator. Pre-snapshot → optional `pacman -Sy` (only when the
  plan touches pacman/AUR) → phases 2..6 in order with per-phase
  `within_phase_key` sort → post-snapshot → `state.toml` history
  commit (CLAUDE.md hard invariant 8: state.toml is the last write).
- **`ConfigWrite` execution** (PR #28): `pearlite_fs::sha256_bytes`
  verifies content, then `pearlite_fs::write_etc_atomic` does the
  write. SHA-256 mismatch surfaces as a typed `ContentSha256Mismatch`
  variant with planned vs actual hashes.
- **Failure path** (PR #30): on mid-pipeline failure, take a
  best-effort post-fail snapshot, build a `FailureRecord` (failed
  action, error message, both snapshot refs), serialize to
  `<failures_dir>/<plan-id>.json`, append a `FailureRef` to
  `state.toml`. All side effects are best-effort and independent so a
  later record-write failure can't mask the underlying `ApplyError`.
- **`Engine::rollback`** (PR #31): looks up the `[[history]]` entry by
  plan_id, dispatches `snapper.rollback(snapshot_pre.id)`. `state.toml`
  is not rewritten — the btrfs revert restores it.

### Week 3 — CLI integration & VM tests

- **`pearlite apply`** (PR #32) — wires `Engine::apply_plan` to clap.
  `RunContext` gains pacman/cargo/systemd/snapper trait objects so
  apply hands them to the engine without per-call construction.
- **`pearlite rollback <plan-id>`** (PR #33) — class 2 / class 3 / no
  retry policy errors mapped to `ROLLBACK_NOT_FOUND` /
  `ROLLBACK_SNAPPER` / `ROLLBACK_STATE`.
- **`pearlite gen list` / `gen show`** (PR #34) — read-only history
  reporters. Tolerant of missing `state.toml` (empty list).
- **`apply --dry-run`** (PR #35) — plan-only path, returns `Plan`
  envelope shape.
- **Plan persistence** (PR #36) — every apply (including `--dry-run`)
  writes the plan JSON to `<plans_dir>/<plan-id>.json`. Closes the
  forensic-trail gap PRD §11 always assumed.
- **`gen show` plan + failure embedding** (PRs #37 / #38) — `data.plan`
  carries the persisted Plan; `data.failure` carries the `FailureRef`
  + parsed `FailureRecord` JSON. `gen show` is now a single-command
  forensic record reader.
- **`apply --plan-file <path>`** (PR #39, ADR-0010) — re-executes a
  persisted plan; strict serde deserialization with no version-shim
  logic. `PLAN_FILE_PARSE_FAILED` catches stale plans from a future
  build via unknown enum variants.
- **VM scenarios** (PRs #40 / #41) — `tests/vm/vm-01` through `vm-06`
  authored as POSIX-sh shellcheck-clean scripts. T4 runner gating
  scaffolded (`scripts/ci/run-vm-tests.sh` requires
  `PEARLITE_VM_TEST=1` for mutating scripts).
- **`apply --prune`** (PR #42, ADR-0011) — forgotten-package removal
  with a configurable threshold (default 5). The diff emits the
  removal actions; the CLI gates the apply.

## What went well

- **The `Action` enum's "one variant per primitive" discipline
  scaled.** Extending the enum's classifier surface (PRs #25, #26)
  needed only one match arm per variant in each new method;
  no dispatch table, no trait-object indirection in the hot path.
- **Trait-first kept the engine apply tests fast.** Every
  `apply_plan` test runs against `MockPacman` / `MockCargo` /
  `MockSystemd` / `MockSnapper` in microseconds; the live adapters
  shipped without any test-side rewrites. The four W1 adapter PRs
  each shipped alongside its mock, so by the time engine apply
  needed them the mocks were already there.
- **PRD §8.2 phase ordering held under refactoring.** `ApplyPhase`
  derives `Ord`; partition into `BTreeMap<ApplyPhase, Vec<&Action>>`
  and iterate in key order — that's the whole orchestrator. Phase
  invariants remained self-evident as features piled on (config
  writes in #28, history commit in #29, failure path in #30, rollback
  in #31).
- **The "no automatic rollback" invariant survived its first real
  test.** PR #30 takes a post-fail snapshot and writes a
  `FailureRecord`, but does **not** invoke `snapper rollback`. PR #31
  ships rollback as a separate operator-driven CLI command. The
  failure of the apply and the recovery from it stayed cleanly
  separated, exactly as PRD §8.5 + CLAUDE.md hard invariant 9
  prescribes.
- **Forensic loop closed end-to-end.** PRs #36 / #37 / #38 form a
  three-step staircase: persist the plan, embed it in `gen show`,
  embed the failure record too. By the time PR #38 landed, an
  operator could recover any apply (success or failure) with one
  command.
- **ADR-blocked Open Questions both resolved during M2.** Plan §14
  flagged `--plan-file` schema stability as needing an ADR; ADR-0010
  picked the strict-deserialization rule. The `--prune`
  drift-threshold was flagged for post-M6; ADR-0011 picked a
  conservative provisional default (5) so M2 W3 could ship the flag
  rather than wait.

## What didn't

- **`pearlite-cli::dispatch_apply` arity climbed.** Too-many-arguments
  hit clippy twice during M2 W3 (PRs #32 and #42). Each time the fix
  was a struct (`ApplyOpts`); each time the struct change cascaded
  through every test helper. Lesson: when a function takes `> 5`
  args of which `≥ 2` are owned `Option<PathBuf>`, reach for the
  options struct on day one. Done now; future flag work lands in
  `ApplyOpts` directly.
- **Schema mismatches in early VM scripts.** Authoring vm-02 through
  vm-06 (PR #41) required two referencing-by-grep fixes where the
  `CargoInventory.crates` map was confused with `installed: BTreeSet`.
  Lesson: when authoring a fixture in a foreign crate, read the type
  definition first, not the test that happens to be open. (The
  scripts didn't actually run on the dev box, so the build error
  caught these — but only because `cargo build` exercises the
  property tests too.)
- **`apply --prune` threshold counter is a heuristic.** It sums every
  `PacmanRemove` + `CargoUninstall` action's package count, which
  today equals the forgotten count because no other code path emits
  those actions. The day a `[remove]` policy emits explicit removes
  (M4-ish), the heuristic stops being right. ADR-0011 §4 flags the
  follow-up; an explicit `forgotten_count` field on `Plan` would be
  the clean fix.
- **One CLI envelope shape break.** PR #38 moved `gen show`'s
  history fields under a nested `data.history` to make room for
  `data.failure`. Pre-1.0 so no consumers broke, but in M5 (CLI
  schema export) we should freeze envelope shapes per command.
- **VM-tier coverage is "syntactically authored" not "executed."**
  vm-01 through vm-06 land as POSIX-sh shellcheck-clean scripts but
  none have run end-to-end on a CachyOS VM during M2. The
  self-hosted runner ADR-009 documents the path; the VM image / CI
  workflow integration is M3 work or later.
- **Solo-dev admin merges still lumpy.** Same M1 observation; same
  outcome (working as intended).
